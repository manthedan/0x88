# tiny_leela repository audit — 2026-05-09

Scope: a single read-only pass over `src/`, `training/`, `scripts/`, `eval/`, `tests/`, `rust/`, `docs/`, repo root, and `package.json`. No code changes were made; this document is the output.

The aim is honest and useful, not exhaustive. Findings are ordered roughly by leverage: the highest-impact items are at the top of each section. A companion file with concrete file/line-level fixes is at [`audit_2026-05-09_quickfixes.md`](audit_2026-05-09_quickfixes.md).

## TL;DR

The project's center of mass is **good**: the goals (browser-deployable Leela-style engine, multi-objective Elo/bytes/latency) are well stated; the model manifest, eval protocol cards, and release-gate framing show real engineering discipline; the TS engine + ONNX runtime + Rust core triple is a sensible substrate. The puffer/MiniZero/lc0 reading is digested correctly.

The center of mass is **not** the problem. The problems are at the edges:

1. **The architecture matrix is growing faster than the gates that would falsify any of it.** Three "active" lanes (CNN, Tactical-MoveFormer, TinyBT/SquareFormer), at least seven distinct `train_*_torch.py` variants, plus AV/rank/regret/risk/uncertainty heads — and no published "X gives +N Elo at fixed compute over Y" headline result. The triage docs even say *don't expand the matrix*; the codebase keeps expanding it anyway.
2. **No automated enforcement of training/inference parity.** The board encoding and policy map are reimplemented in Python (`fixed_policy_moves`, `compact_tokens_to_residual_planes`, four different `PIECES =` strings) and TypeScript (`POLICY_MOVES`, `onnxInputPlanes`, `squareformerCompactInput`), with no tests that compare them on a shared corpus. This is the single most likely place for silent training/inference drift to hide.
3. **The Python training tree has near-zero shared-library structure.** 31 standalone scripts, each with its own copy of helpers; `train_residual_av_multicache_torch.py` defines an entire `Net` class on one ~600-character line. Hostile to review, refactor, and bug isolation.
4. **The TS PUCT batched path has a subtle Q-pollution bug** (zero-value backups when `selectLeaf` lands on an already-expanded leaf) and a global mutable `dataStride` in `squareformerEvaluator.ts` that is module-level state. Not catastrophic, but real.
5. **The repository is hauling around dead weight.** `awscliv2.zip` (70 MB), `tiny-leela-netlify-sfaux-512.zip` (74 MB), and `Nibbler.standalone.html` (2 MB, committed) sit at the root; `tiny_leela_research.md` exists in two copies; `dovetail.*.jsonl` from a prior autoresearch loop is committed.
6. **60 markdown docs, 23,665 lines total, with five 1000+-line brainstorms.** `docs/README.md` admits it. The "do not delete" cleanup policy is the bug, not the safeguard.

The single most cost-effective intervention is **freeze the architecture matrix, add cross-language parity tests, and publish one absolute-anchor-Elo number per active model**. Everything else flows from those.

## 1. Project direction and methodology

### 1.1 The architecture matrix is wider than the evaluation funnel

The `expert_handoff_2026-05-current.md` packet is honest about this: three lanes, all "active," and the open question is "which architecture should become the next browser champion." But the codebase reflects more lanes than the doc claims:

```
training/train_residual_torch.py
training/train_residual_aux_cache_torch.py
training/train_residual_aux_multicache_torch.py
training/train_residual_av_multicache_torch.py
training/train_residual_sideinfo_multicache_torch.py
training/train_board_cnn.py
training/train_squareformer_torch.py
training/train_squareformer_v2_torch.py
training/train_squareformer_hybrid_av_multicache_torch.py
training/train_channelformer_av_multicache_torch.py
training/train_moveformer_cnn_av_torch.py
training/train_moveformer_kitchensink_torch.py
training/train_squaremoveformer_kitchensink_torch.py
```

The `model_manifest.current.md` lists 17 models with 4 distinct "Family" labels (`ResidualCNN`, `ResidualCNN_AV`, `ResidualCNNHybrid`, `MoveFormer_CNN_AV`, `CNN_ChannelFormer`, `SquareFormer`). One is tagged `current-search-champion` (`cnn_80x5_100m_e3`), but no row in the manifest contains an Elo number — strong or weak, anchor-relative or pool-relative. The arena/anchor JSONs exist, but the manifest doesn't surface them.

`docs/deepresearch_architecture_triage_2026-05.md` opens with: *"The DeepResearch report did not change the active roadmap... Do not use this report to restart the architecture plan or expand the experiment matrix."* The very next paragraphs then list six new directions (search-light framing, geometry/history bias, move-query decoder, regret/risk metrics, conditional PUCT, h7/h8 history). The intent is right; the discipline isn't.

**Recommendation.** Pick a small kill-criteria contract for each lane and write it into the manifest. Concretely:

- *CNN baseline* — incumbent, no kill criteria; this is the floor.
- *CNN-AV* — must show ≥ +30 Elo over equal-budget classic-PUCT CNN at 64 visits in the next anchor arena, or AV head training is paused (not deleted) and the cache build is throttled.
- *MoveFormer* — already "parked" (per `head_ablation_roadmap.md`); confirm the park is real by removing the queue/cache scripts from `package.json`.
- *SquareFormer-V2-AV* — must show ≥ baseline CNN policy CE *and* ≥ baseline AV-MSE on a shared dev set at equal latency, or shrink to a single 1M smoke and stop new V2/V3 plumbing work.

Today, none of these lines are drawn in the manifest. They are all "active" forever.

### 1.2 No absolute-strength signal

The eval-process doc is the best document in the repo. It correctly insists on protocol cards, paired pentanomial outcomes, anchor pools, SPRT, and conditional Elo. But:

- `OpenBench adoption is blocked on a stable Tiny Leela UCI wrapper` (`expert_handoff_2026-05-current.md` line ~219). `scripts/uci_tiny_leela.mjs` exists (197 lines). It either works or it doesn't — finishing it is a tractable concrete task, and is probably the single highest-leverage open item.
- `eval/uci_anchor_arena.mjs` runs vs. Stockfish/Maia anchors locally. The anchor-arena outputs live in `artifacts/anchor_arena/...`, but they don't roll up into the manifest.
- `dev_policy_top8` and the `composite` metric (`policy_ce + 0.25*wdl_ce + 2.0*av_mse`) are the de facto promotion signals, even though the eval-process doc explicitly warns against this kind of single-scalar offline-only signal.

**Recommendation.** Promote one anchor-arena number into `model_manifest.current.md`, even if it's noisy. Something like *"Stockfish UCI_Elo 1320 score-rate, 32 visits, 20 paired UHO openings, Wilson 95% CI"*. This is a 50-line aggregator change, and it changes the project's center of gravity from "the offline metrics suggest…" to "this model wins or loses against this anchor at this protocol."

### 1.3 Self-play infrastructure is over-designed before being launched

`docs/distributed_selfplay_training_system_design.md` is 1,882 lines. Its first paragraph correctly says: *"Do not launch full distributed self-play until supervised/AV/search gates, UCI wrapping, and local promotion tests are stable."* `docs/puffer_selfplay_infra_plan.md` (~600 lines) covers a smaller "ChessVecWorker" iteration. `scripts/selfplay_generate.mjs` exists and works.

The pattern: very large design doc, small actually-running implementation, and the gap between them is "future architecture / reference." That's fine as a one-time investment, but the project has *two* such docs, and the proximate work (UCI wrapper, anchor-arena rollup, AV-PUCT promotion gate) is more important.

**Recommendation.** Mark both self-play docs as `Frozen until Q3` with one sentence at the top: *"Re-open this when X concrete prerequisite is true."* That signals intent and stops the docs from accreting.

### 1.4 Triage docs as a methodology pattern

There are two recent triage docs (`deepresearch_architecture_triage_2026-05.md`, `unsloth_rl_economics_triage_2026-05.md`). Each is multiple hundreds of lines, both end with *"this did not change the roadmap."* That is the ideal triage outcome — but a triage doc that takes a multi-hundred-line read to communicate "no change" is itself a form of accretion.

**Recommendation.** Adopt a 1-paragraph triage template at the top of each such doc:

```
Verdict: NO CHANGE / SMALL CHANGE / RESTART
Adopted: <one-line bullet list>
Rejected: <one-line bullet list>
Open question carried back into the roadmap: <single sentence>
```

Everything below the template is reference material readers can skim or skip.

### 1.5 Two engines (TS + Rust), one parity script

The Rust core (`rust/tiny_leela_core/src/lib.rs`, 708 lines) duplicates movegen, makeMove, and `search_root` from the TS engine. `scripts/compare_rust_ts_search.mjs` and `scripts/compare_puct_trace.mjs` exist as parity tools, plus an `arena:rust-ts` npm script. None of them run as part of `npm test`.

This is the canonical setup for silent drift. Today's TS movegen treats the en-passant square as set unconditionally on any double pawn push; Rust's may or may not do the same — and a difference there flips the action-id of one position in some FENs, silently corrupting any cache built from one engine and read by the other.

**Recommendation.** Add a tiny perft test (under one second) to `npm test` that calls *both* engines and asserts equality at depths 1–3 from a list of 5–10 representative FENs (including en-passant edge cases, castling, promotions). It pays for itself immediately.

### 1.6 The Muon experiment is a model for handling negative results

`docs/muon_experiment_notes.md` is excellent: setup, table, late-stage smoke, interpretation, "we still do not have a win for Tiny Leela yet." This is exactly the right shape for a research note. It would be more useful if its conclusion ("Muon is parked, AdamW remains default") were wired into:

- the `--optimizer muon` flag in `train_residual_av_multicache_torch.py` being marked deprecated, *or*
- a top-level `docs/decisions/` log entry that survives the next round of architecture churn.

**Recommendation.** Adopt a short `docs/decisions.md` ADR-style log: *date / decision / one-sentence rationale / superseded-by*. Five lines per decision. This is where "we tried Muon, it lost, we kept the codepath gated" lives.

## 2. Code-level findings (TypeScript)

### 2.1 PUCT batched path: zero-value backup on duplicate-leaf hits

`src/search/puct.ts` line 329:

```ts
if (sel.node.expanded) { prepared.push({ sel, value: 0 }); continue; }
```

This handles the case where `selectLeaf` lands on an already-expanded node within the same batch. Within a single batch, `selectLeaf` only stops at expanded nodes if their `terminalValue !== null` (the prior branch handles that). The case where `expanded === true` and `terminalValue === null` should be unreachable in clean execution, but if it ever happens (race, future refactor, deduplicated leaf), this code backs up `value: 0` — biasing Q estimates of every edge along that path toward 0.

**Fix.** Either assert this is unreachable, or skip the backup entirely (decrement virtual visits, do not increment `visits` or `valueSum`, don't count in `completedVisits`). Backing up zero is the silently-wrong option.

### 2.2 PUCT batched path: silent default when `evalSlot` is undefined

Same file, line 358:

```ts
searchPolicy.backup(item.sel.path, item.value ?? values[item.evalSlot ?? 0] ?? 0, context);
```

If `evalSlot` is `undefined` and `item.value` is also `undefined`, this picks `values[0]`. That is reachable when the prepared item is a "duplicate-leaf" or "expanded fallback" item where `value` was set but to undefined-flavored zero, and `evalSlot` happens to coincide with slot 0. It's defensive, but it papers over class-of-bug rather than asserting structure.

**Fix.** Make `prepared` a discriminated union: `{ kind: 'terminal', value } | { kind: 'duplicate' } | { kind: 'evalSlot', slot }`, and back up by case.

### 2.3 PUCT recursive path inconsistent with batched path

`simulate` (line 255) does not touch `virtualVisits`, while `selectLeaf` (line 270) increments them. Both are reachable depending on `batchSize`. Today `simulate` only runs at batchSize=1, where virtualVisits don't matter, but anyone reading the code has to derive that from context. The two paths should share a selection step.

**Fix.** Either inline `simulate` into `runBatchedVisits` (single batched/sequential implementation, batchSize=1 falls through), or make `simulate` use `selectLeaf` and discard the virtualVisits at the end.

### 2.4 `squareformerEvaluator.ts` module-level mutable `dataStride`

Lines 41–53:

```ts
function addBoardFeatures(data: Float32Array, board: BoardState, boardIndex: number, planesPerBoard: number) {
  const base = boardIndex * planesPerBoard;
  for (let sq = 0; sq < 64; sq++) data[sq * dataStride + base + pieceId(board.squares[sq])] = 1;
}

let dataStride = 0;

function squareformerFloatInput(...) {
  ...
  dataStride = inputDim;
  ...
  addBoardFeatures(data, board, 0, planesPerBoard);
  ...
}
```

`dataStride` is a module-level mutable global that `addBoardFeatures` reads. The compact (`int64`) path reads it implicitly via passed `stride`; the float path uses the global. This is a hidden side channel between two functions in the same module. It is not thread-safe (no JS threading today, but workers exist), and it traps any reader who tries to call `addBoardFeatures` independently.

**Fix.** Pass `stride` explicitly into `addBoardFeatures` like the compact path does.

### 2.5 Onnx evaluator: positional fallback and `?? -1` indexing

`src/nn/onnxEvaluator.ts`:

- Line 209: `const policyRaw = (outputs.policy_logits_legal?.data ?? Object.values(outputs)[0].data) as Float32Array;` — positional fallback by output ordering. If a future export adds a new tensor at slot 0, this silently grabs the wrong tensor.
- Line 47 (and `studentEvaluator.ts:289`): `legal.reduce((sum, move) => sum + (probs[policyIndex(move) ?? -1] ?? 0), 0)` — uses JS's lax `array[-1] === undefined` to absorb missing-policy-index cases. Works, but obscure.

**Fix.** Insist on named outputs with a clear error if missing; replace `?? -1` with an explicit `if (index === undefined) continue;`.

### 2.6 Movegen perf is the engine's wall-clock floor

`src/chess/movegen.ts:138`:

```ts
export function legalMoves(board: BoardState): Move[] {
  return pseudoLegalMoves(board).filter((move) => !inCheck(makeMove(board, move), board.turn));
}
```

For each pseudo-legal move:

1. `makeMove` clones the entire 64-element `squares` array (`cloneBoard`).
2. `inCheck` scans for the king with `findIndex` (O(64)) and calls `isSquareAttacked` (8 ray scans + knight + king + pawn templates).

Plus `puct.ts:225` builds child history with `[boardToFen(parent.board), ...parent.historyFens]` — `boardToFen` is a string concatenation of all 64 squares with rank breaks and metadata. Per visit per child.

For policy-only or 32-visit search the cost is acceptable. For 256-visit AV-PUCT batches the TS engine becomes the bottleneck — and `arena_rust_ts.mjs` exists exactly because of this.

**Fix priority.** Low-effort wins, in order:

1. Cache `kingSquare` per `BoardState` (lazy + invalidated on `makeMove`).
2. Stop materializing `historyFens` at every child — pass a small fixed-length ring of FEN-strings or `BoardState` references.
3. Replace `cloneBoard` with a typed-array squares representation. The Rust core already does this; mirror its `[Option<Piece>; 64]`/u8 layout in TS as a `Uint8Array(64)`.
4. Move the legal-filter to a "make + unmake" pattern so the clone goes away entirely. Optional, biggest payoff.

If the Rust engine is the canonical path for any serious search, these become moot — but `package.json` still has `selfplay:generate` pointing at the TS path.

### 2.7 Two redundant TS evaluator paths

`src/nn/studentEvaluator.ts` carries:

- a hand-rolled JS conv (`convReluResidual`, lines 155–170),
- a deterministic `stableWeight` RNG to seed conv params for "frozen_conv_*" students (lines 81–119),
- a tiny per-board conv-features path (`convStudentFeatures`, lines 185–236),

all alongside the modern `OnnxEvaluator`. These predate the ONNX runtime path. The interface enum (`'linear_fen_student' | 'frozen_conv_fen_student' | 'frozen_conv_feature_mlp_student' | 'tiny_board_cnn_student' | 'tiny_board_residual_student'`) shows the archaeology.

**Recommendation.** When (not if) any of those legacy `kind` values is no longer used by any tracked artifact, delete the corresponding path. Don't carry them indefinitely.

### 2.8 Action-id encodings: three coexisting schemes

The codebase uses (at least) three move-encoding schemes:

- `moveToActionId` from `moveCodec.ts`: `(from*64 + to)*5 + promo` — 20480 entries, used by `src/search/puct.ts` and the `Evaluation.policy` map.
- `POLICY_MOVES` from `policyMap.ts`: 1968 unique UCI strings (queen/knight/promo set), used by the residual CNN policy head and the `OnnxStudentMeta.policy_map === 'uci_queen_knight_promo_v1'` check.
- ChessBench compact 20480: `from*64 + to` for non-promotions, `4096 + from*64*4 + to*4 + promo` for promotions (`onnxEvaluator.ts:54`, `squareformerEvaluator.ts:127`). Different layout from `moveToActionId`.

This is fine if it's documented, and it is — partially — in `docs/move_encoding.md`. But the cross-encoding mappings live as one-off helpers in evaluator modules, not in a single canonical conversion table. This is exactly the place where a "wrong direction of conversion" bug will eventually be introduced. (Already there's a mirror-FEN consistency test (`tests/mirrored_fen_consistency.test.mjs`, 10 lines) — keep extending that to all three encodings.)

**Recommendation.** Promote a single `src/chess/moveEncodings.ts` with `toActionId(move)`, `toPolicyIndex(move)`, `toChessBenchAvClass(move)`, and `toSquareformerPolicyIndex(move)`. Then write one parity test that asserts inverse round-trips for all 20480 candidate moves (not just legal ones in a few positions).

## 3. Code-level findings (Python training and data pipeline)

### 3.1 Massive copy-paste, no shared library

Across `training/*.py` (31 files):

- `PIECES = ...` is defined in 15 files in 4 distinct forms: `'PNBRQKpnbrqk'` (no leading dot, single quotes), `"PNBRQKpnbrqk"` (no dot, double quotes), `'.PNBRQKpnbrqk'` (with leading dot, single), `'.PNBRQKpnbrqk'` (with dot, no spaces). The leading-dot variant means "empty square = piece id 0"; the no-dot variant uses 0..11 as the piece index and shifts states by 12. Use the wrong one and an entire training run silently encodes the empty-square plane as a knight. There is no shared test that asserts every script's `PIECES` agrees.
- `fixed_policy_moves` is duplicated verbatim in `train_residual_torch.py` and `train_board_cnn.py`. It also has a sibling implementation in `src/chess/policyMap.ts:buildPolicyMoves`. None of these are tested against each other; only the TS one has `assert.equal(POLICY_SIZE, 1968)`.
- `expand_collection`, `open_av_cache_dir`, `compact_tokens_to_residual_planes`, `decode_move_classes_np` appear (with minor variants) across at least 7 `train_*_multicache_torch.py` scripts.
- The only sign of cross-script reuse is one chain: `train_residual_aux_multicache_torch.py` (and a few siblings) imports from `train_residual_torch`, and `train_squaremoveformer_kitchensink_torch.py` imports `FEATURE_NAMES, move_feature, ...` from `build_moveformer_sidecar_cache`. Importing implementation details from one ad-hoc script into another is the worst kind of reuse — it locks the importer to the importee's argparse defaults and helper layout.

**Recommendation.** Carve out `training/_lib/encoding.py`, `training/_lib/caches.py`, `training/_lib/optim.py`. Move the duplicated helpers there. Add `tests/test_encoding_parity.py` that imports both the Python `fixed_policy_moves` and parses the TS `POLICY_MOVES` (or a JSON dump of it) and asserts equality. This is two days of work and removes a permanent footgun.

### 3.2 Single-line model classes

`training/train_residual_av_multicache_torch.py:131`:

```python
class Net(nn.Module):
    def __init__(self):
        super().__init__(); ch=args.channels; self.stem=nn.Conv2d(C,ch,3,padding=1); self.blocks=nn.Sequential(*[Block(ch) for _ in range(args.blocks)]); pin=ch*64+(ch if args.policy_head=='hybrid' else 0); self.policy=nn.Linear(pin,P); self.wdl=nn.Linear(ch,3); self.promo_emb=nn.Embedding(5,ch); self.av=nn.Sequential(nn.Linear(ch*4,ch),nn.GELU(),nn.Linear(ch,1)); self.rank=nn.Sequential(nn.Linear(ch*4,ch),nn.GELU(),nn.Linear(ch,1)) if args.separate_aux_heads else None; self.regret=nn.Sequential(nn.Linear(ch*4,ch),nn.GELU(),nn.Linear(ch,1)) if args.separate_aux_heads else None
```

That's the entire `__init__`. A `pdb` line break inside, a missed conditional, or a typo in `if args.separate_aux_heads else None` is invisible. The other `train_*_torch.py` files follow the same convention. Loss/training/eval lines look the same:

```python
xb,mv,val,reg,mask=av_batch(); _,_,h=net(xb); sc=net.av_scores(h,mv); rank_sc=net.rank_scores(h,mv) if args.separate_aux_heads else sc; av=F.smooth_l1_loss(torch.tanh(sc.float())[mask],val[mask]); rank=F.cross_entropy(rank_sc.float().masked_fill(~mask,-1e9), val.masked_fill(~mask,-1e9).argmax(1)); preg=(net.regret_scores(h,mv).float() if args.separate_aux_heads else (sc.float().max(1,keepdim=True).values-sc.float())).masked_select(mask); rloss=F.smooth_l1_loss(preg,reg.masked_select(mask)); loss=args.av_weight*av+args.rank_weight*rank+args.regret_weight*rloss
```

This style makes diffs (and code review) extremely lossy. PR diffs see an entire reformatted line as a single change rather than a delta on one nested expression. It is also markedly worse for the human writing the code, which has knock-on effects on bug rate.

**Recommendation.** Run `ruff format` (or `black` with line length 100) over `training/`. Disable the line-density rules. This is one PR. The model definitions become reviewable.

### 3.3 Observability is `print METRIC k=v`

The training scripts emit lines like:

```
METRIC step10000_dev_policy_ce=2.83 dev_av_top1=0.18 ...
```

External orchestration (the `dovetail.*` files at repo root suggest an autoresearch loop) presumably regex-scrapes these. There's no structured JSONL-per-step file, no checkpoint metadata schema test, no schema versioning. The format is also inconsistent across scripts: some emit `progress epoch=X step=Y`, some emit metrics with `_` separators, some emit `loss=` keys.

**Recommendation.** Add a tiny `training/_lib/metrics.py:emit(label, **kv)` helper that writes both a `print METRIC` line *and* a JSONL row to `args.metrics_jsonl_out` if set. Then anything that wants stable observability can read the JSONL.

### 3.4 Memmap fancy-indexing storms

`refill_policy_pool` has a comment: *"Force a real ndarray copy: random fancy-indexing inside memmaps causes page-fault storms."* Good that it's documented; bad that it's documented in three different scripts independently. This is exactly the kind of helper that belongs in `training/_lib/caches.py`.

### 3.5 Subprocess use of `zstd`

`scripts/build_supervised_dataset_streaming.py` shells out to `zstd -dc` for `.zst` files. If `zstd` is missing, it fails late with a `FileNotFoundError` from `subprocess.Popen`. `requirements-onnx.txt` exists but does not pin `pyzstd`/`zstandard`. This is a real environment-portability hazard, especially across the AWS Batch CPU spot images you're now spinning up.

**Recommendation.** Either add `zstandard` to `requirements-onnx.txt` and use it in-process, or add an explicit `which zstd || exit` check in the bootstrap scripts.

### 3.6 The "no Python tests" hole

```
tests/train_student.test.mjs  (34 lines, smoke test of train_student.py)
```

That is the entire Python test surface. None of the AV-cache builders, AV-trainers, V2 SquareFormer trainers, or ONNX exporters have any test coverage. The TS test suite is decent (8 PUCT-core invariants, evaluator parity, ONNX history, etc.). The Python side is bare.

**Recommendation.** Even one test per category is enormously valuable:

- `test_compact_tokens_to_residual_planes_roundtrip.py` — fixed dummy input, fixed expected output.
- `test_action_value_cache_meta.py` — open a tiny synthetic AV cache, assert shape/dtype contract.
- `test_export_onnx_smoke.py` — train for one batch on synthetic data, export ONNX, load with onnxruntime, assert shape.

These add ~100 lines and make the next refactor not terrifying.

### 4.3 `artifacts/` is 37 GB, `data/` is 1.2 TB

Both are gitignored, which is correct. But there's no documented retention/rotation policy. `artifacts/` has 208 directories, many of them smoke-test outputs. A 50-line `scripts/prune_artifacts.py --older-than 30d --dry-run` would be cheap insurance against a "disk full" surprise on the actual training box.

### 4.4 `package.json` script names mix conventions

There are 38 npm scripts. Some use `:` namespaces (`eval:phase-b`, `selfplay:generate-atomic`), some use `-` (`bench-movegen` would be `bench:movegen` in the namespace style — and is). Some are wrappers around `.venv-onnx/bin/python ...`, some shell out to `bash scripts/*.sh`. This grew organically.

Not a bug, but the surface area is now large enough that a reader can't tell from name alone whether a script is a smoke test, a long-running queue, or a one-shot validator. A 10-line `docs/scripts.md` table would help.

### 4.5 `tsconfig.json` with `"noEmit": true`

`tsconfig.json` sets `"include": ["src/**/*.ts"]` and `"noEmit": true`. The actual runtime is Node with `--experimental-strip-types`. Fine, but it means no `tsc --noEmit` check is wired into `npm test`. Type errors in `src/` survive until they crash at runtime.

**Recommendation.** Add `"typecheck": "tsc --noEmit"` to `package.json` and run it from `npm test`.

## 8. Closing thought

The biggest force in this codebase is curiosity — the willingness to try MoveFormer, ChannelFormer, SquareFormer V0/V1/V2/V3, CNN-AV, hybrid heads, h2/h7/h8 caches, Muon, smolgen-lite, action-values from ChessBench top-8 *and* top-48, conditional PUCT, Gumbel-root, AV/rank/regret/risk/uncertainty heads, all in one repo. That is the strength of the project, and it should not be eliminated. But the matching counterforce is missing: a small, mechanical filter that says *"this idea/model/cache/training run failed the falsification gate, retire it."* The triage docs are the closest thing to that filter, and they are too gentle.

If you adopt one habit from this audit, make it: **every active lane has a written kill-criterion, and the kill-criterion is checked monthly on the manifest.** The existing eval-process and release-gate docs already describe how to evaluate models honestly; what's missing is acting on those evaluations as decisively as the project acts on starting new ones.
