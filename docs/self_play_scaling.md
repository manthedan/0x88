Yes. These architectures **can scale into lc0-style self-play**, but only if we stop thinking of the model as “a supervised imitator” and make it part of a **policy-improvement loop**.

The right mental model is:

```text
supervised distillation:
  teacher search → model

self-play improvement:
  model → search → improved labels → stronger model → stronger search → ...
```

Supervised learning from a fixed oracle is mostly bounded by that oracle’s coverage and quality. But AlphaZero/lc0-style learning creates a moving oracle: **current model + search**. The network alone may be weaker than its teacher, but **search with the network can be stronger than the raw network**, and training compresses that search-improved behavior back into the next network. Lc0’s own primer describes the neural net as outputting policy and value, and PUCT/MCTS as the second component that uses those outputs to search; lc0’s training docs describe self-play games as the input data used to improve the neural network. ([Leela Chess Zero][1])

The short answer is: **yes, the SquareFormer/action-value/DeepSeek-inspired architectures can scale to self-play**, but I would structure the project as **search-light first, AlphaZero-like self-play second, targeted self-play third** rather than jumping straight into full lc0-scale RL.

---

## 1. Why these architectures are compatible with self-play

For lc0-style self-play, a model needs to expose at least this interface:

```text
fθ(position) →
  policy over legal moves
  value / WDL
```

Our proposed models can do that:

```text
SquareFormer:
  policy_head → from-to move policy
  WDL/value_head → position value

SquareFormer-AV:
  policy_head → candidate moves
  WDL/value_head → leaf value
  action_value_head → move reranking / search acceleration

DeepSeek-inspired version:
  same heads
  plus uncertainty, multi-ply auxiliary heads, deployment-aware quantization,
  and multi-teacher/on-policy distillation machinery
```

So the architecture does not block self-play. In fact, the search-light proposal already assumes policy, WDL/value, action-value, and uncertainty heads, with conditional PUCT for difficult positions. That can be turned into a full self-play actor simply by letting PUCT produce the improved training target. 

The DeepSeek-inspired notes also fit naturally: specialist-to-student distillation, on-policy data generation, fault-tolerant rollouts, quantization-aware training, and multi-ply auxiliary heads all become useful once the student starts producing its own games.

## 2. The core self-play loop

The lc0/AlphaZero-style loop is:

```text
1. Start with model fθ.

2. Use fθ + PUCT to play games against itself.

3. At each move, store:
   - position
   - legal move mask
   - search visit distribution
   - search value / root WDL
   - final game result
   - optional extra labels

4. Train fθ+1:
   - policy learns to imitate search visits
   - value learns to predict game result / WDL / search value

5. Evaluate fθ+1 against fθ.

6. If stronger, promote fθ+1 as the new self-play generator.

7. Repeat.
```

This is the key difference from plain supervised learning. The model is not just copying a fixed teacher. It is learning from a **policy-improvement operator**:

```text
raw network policy
  → PUCT search
  → improved policy target
  → next network
```

Lc0’s public network/run documentation describes this reinforcement learning pattern: use a network to generate games, train the next version from those games, then generate games with the updated network; it also notes that official lc0 runs use RL, while supervised runs can retrain on existing data with different training choices. ([Leela Chess Zero][2])

---

## 3. Why this can improve beyond the supervised oracle

A fixed supervised oracle gives you:

```text
position → teacher answer
```

If the teacher is wrong, shallow, biased, or absent in some part of the state space, the model is limited.

Self-play gives you:

```text
position distribution generated by the current model
+ search-improved targets on exactly the positions the model reaches
+ final outcomes from actual play
```

That matters because the model starts learning from its own weaknesses. If it repeatedly enters a class of positions it misunderstands, the self-play/search loop can generate dense data there.

The improvement mechanism is:

```text
network suggests moves
search corrects or sharpens them
training absorbs the correction
new network makes search better
new search finds better corrections
```

This does not guarantee infinite improvement. It can plateau, collapse, overfit, or reinforce blind spots. But it is the mechanism by which AlphaZero/lc0-style systems can go beyond static imitation.

## 4. The best structure for our project

I would use a **three-phase self-play curriculum**.

### Phase A — supervised/distilled bootstrap

Do not start from random. Start with the strongest cheap prior we can get.

```text
Data:
  Stockfish/lc0 teacher labels
  human/Maia data if we want human-like calibration
  puzzles/tactics
  tablebase endgames
  opening diversity

Train:
  policy
  WDL/value
  action-value top-k head
  uncertainty head
  multi-ply auxiliaries
```

Goal:

```text
Get a non-random engine that can already play legal, coherent chess.
```

This phase is still supervised, but it gives self-play a good starting point. Starting from random is philosophically pure but compute-wasteful.

---

### Phase B — search-improved self-play

Now use the model itself as the actor, but with search.

```text
Actor:
  latest accepted model
  PUCT search
  temperature / root noise for early moves
  opening diversity

Per move, store:
  position
  legal mask
  raw network policy
  PUCT visit distribution
  root WDL/value
  selected move
  search depth/nodes
  policy entropy
  uncertainty estimate
  final game result
```

Training target:

```text
policy_target = normalized PUCT visit counts
value_target  = final WDL/game result, optionally mixed with root WDL
```

This is the canonical lc0-style step. Lc0’s overview describes the engine as loading a neural network and running MCTS, while the RL loop infrastructure supports training/testing/deployment around self-play and neural-network updates. ([Leela Chess Zero][3])

A good first loss:

```text
L =
  KL(π_search || π_model)
+ CE(WDL_result_or_search, WDL_model)
+ CE(value_bucket_target, value_bucket_model)
+ λ_aux * auxiliary_losses
+ weight_decay
```

The important point is:

```text
Train on the search-improved policy, not just the move that was played.
```

The full visit distribution carries much more information than a one-hot move.

---

### Phase C — targeted self-play / on-policy improvement

Once the loop works, do not generate self-play uniformly. Target the positions where improvement is most likely.

Prioritize positions with:

```text
high policy entropy
high uncertainty
large disagreement between policy and action-value head
large disagreement between current model and teacher
large value swing during search
failed puzzles
blunders in evaluation games
rare endgames
sharp tactical positions
model-vs-model losses
```

This is the chess equivalent of DeepSeek-style on-policy distillation: the student generates its own hard distribution, then stronger or more expensive processes label it. 

A practical targeted loop:

```text
1. Student self-plays cheaply.
2. Mine hard positions.
3. Reanalyze hard positions with:
   - deeper PUCT
   - Stockfish
   - stronger lc0
   - tablebases
4. Train on those high-value labels.
5. Promote only if match testing improves.
```

This is probably the most compute-efficient route for a scrappy project.

## 5. How the action-value head fits self-play

The action-value head is one of the most valuable additions for self-play.

In supervised/search-light mode, it does:

```text
position + candidate move → predicted value of that move
```

In self-play, PUCT can generate training labels for it.

At a root position, PUCT accumulates per-move stats:

```text
move m:
  N(m) = visit count
  Q(m) = average backed-up value
  P(m) = prior
```

So we can train:

```text
policy_head(position)        → visit distribution N / sum(N)
value_head(position)         → final result / root WDL
action_value_head(pos, move) → Q(m) or value bucket for move m
```

This is very useful because it compresses some of PUCT’s work into the model.

At inference time:

```text
policy proposes top-k moves
action-value head reranks them
PUCT, if used, starts from a much better move ordering
```

During self-play:

```text
better action-value head
→ better root move ordering / better search allocation
→ stronger self-play data
→ better next action-value head
```

This becomes a second policy-improvement channel.

I would not train action-value on every legal move at first. Store labels for:

```text
top PUCT moves
high-policy moves
checks/captures/promotions
random legal distractors
moves with surprising high/low Q
the played move
```

That gives the model contrastive knowledge without exploding labeling cost.

## 6. How multi-token prediction maps to multi-ply auxiliary heads

The DeepSeek-inspired “multi-token prediction” analogy is very strong for chess. In language, multi-token prediction asks the model to predict more than the next token. In chess, the analogue is:

```text
predict more than the next move
```

But we should be careful: the main goal is still strength, not generating pretty PVs. Multi-ply heads should be **auxiliary training signals** that make the trunk learn tactical and opponent-response structure.

Good multi-ply auxiliary targets:

```text
next_move_policy:
  π_search at current position

reply_policy:
  π_search at the position after the selected/top move

second_own_move_policy:
  π_search two plies later along the search PV or played game

PV tokens:
  best line from root search: m1, m2, m3, m4...

afterstate_value:
  WDL/value after top candidate moves

threat_head:
  opponent’s best reply value after our candidate move

tactical_flag_head:
  whether the line contains check, capture, promotion, mate threat, etc.
```

A concrete loss:

```text
L =
  KL(π0_search || π0_model)
+ CE(WDL0_target, WDL0_model)
+ CE(value_bucket0, value_bucket0_model)
+ λ_av  * AV_loss(position, candidate_moves)
+ λ_r1  * KL(reply_policy_search || reply_policy_head)
+ λ_pv  * CE(PV_move_1..K)
+ λ_v1  * CE(afterstate_value_bucket)
+ λ_unc * uncertainty_calibration_loss
```

Why this helps:

```text
policy head:
  learns what to play now

reply head:
  learns what the opponent will likely do

second-move/PV heads:
  force the trunk to represent short tactical continuations

afterstate value:
  connects candidate moves to consequences

uncertainty:
  tells runtime when to spend search
```

This is not a replacement for PUCT. It is a way to make each model evaluation contain more implicit planning.

## 7. Self-play should not be “always full PUCT”

For our goal — strongest lightweight neural chess engine — I would not make “search-less” the religion, and I would not make “always heavy PUCT” the religion either.

Use a **compute ladder**:

```text
Mode 0: policy-only
Mode 1: policy + action-value rerank
Mode 2: small PUCT, 16–64 nodes
Mode 3: deeper PUCT, 128–512 nodes
Mode 4: teacher reanalysis / offline deep search
```

During training, generate data at multiple levels.

Example:

```text
70% cheap self-play:
  policy + action-value + 32-node PUCT

20% hard-position self-play:
  128–512-node PUCT

10% teacher reanalysis:
  Stockfish/lc0/tablebase on mined positions
```

This has two benefits:

```text
1. It matches deployment.
   The model learns from the kind of search it will actually use.

2. It spends compute where it matters.
   Easy positions do not need huge search.
```

The earlier search-light design explicitly recommends conditional search based on uncertainty rather than pure searchlessness, which is exactly the structure I would carry into self-play.

## 8. A concrete self-play data record

Every stored training position should be rich enough to support future heads.

```json
{
  "run_id": "sp_0007",
  "net_id": "sqf_0064",
  "fen": "...",
  "side_to_move": "w",
  "legal_moves": ["e2e4", "d2d4", "..."],

  "raw_policy": {
    "e2e4": 0.21,
    "d2d4": 0.18
  },

  "search_visits": {
    "e2e4": 512,
    "d2d4": 301

  },
```

Even if v1 only uses policy/WDL, storing richer data lets v2 train action-value, reply, PV, and uncertainty heads without regenerating everything.

## 9. Replay buffer design

A naive replay buffer can poison the loop. I would use a mixture:

```text
recent self-play:
  keeps training on current on-policy distribution

best historical self-play:
  prevents forgetting

teacher-labeled hard positions:
  prevents self-play blind spots

tablebase/endgame positions:
  guarantees exact endgame grounding

opening-diverse positions:
  prevents overfitting to common openings

loss positions:
  learn from the model’s own failures
```

Example sampling mix:

```text
50% recent accepted-net self-play
20% hard mined positions
10% teacher reanalysis positions
10% tablebase/endgame curriculum
10% opening/puzzle/tactical curated data
```

As strength improves, shift toward:

```text
more on-policy hard positions
less generic supervised data
```

But I would never remove teacher/tablebase anchors entirely. They act as guardrails.

## 10. Promotion and evaluation

Do not let every trained checkpoint generate future data. That can cause regressions to compound.

Use a promotion gate:

```text
candidate net vs current best
fixed openings
fixed nodes/search settings
both colors
enough games for confidence
promote only if candidate is stronger or meaningfully better on target metrics
```

For a small project, start simple:

```text
quick gate:
  400–1000 games, fixed nodes, broad opening set

serious gate:
  5000+ games, multiple search budgets

release gate:
  browser latency + Elo + puzzle + WDL calibration
```

Track not only Elo:

```text
policy KL
action-value ranking
WDL calibration
puzzle accuracy
endgame accuracy
fixed-node Elo
browser eval/sec
fallback frequency
```

This matters because a model might gain policy-only strength but lose search compatibility, or gain Elo only by becoming slower.

## 11. How to structure exploration

Self-play fails if it only repeats what the current model already likes.

Use exploration at the root:

```text
temperature for early moves
Dirichlet/root noise or equivalent
opening randomization
UHO/opening suites
sampling from safe candidate moves
```

But reduce randomness later in the game:

```text
opening:
  more exploration

middlegame:
  moderate exploration

endgame:
  lower exploration / tablebase if available
```

Also use **targeted starts**, not only the normal starting position:

```text
random openings
known tactical motifs
imbalanced material positions
endgame tablebase positions
positions where previous model blundered
positions with high teacher disagreement
```

If every self-play game starts from the initial position, you waste massive compute on familiar openings.

## 12. Can a search-less architecture still improve through self-play?

Yes, but with a caveat.

A purely policy-only model has a weak improvement operator:

```text
model plays games
final result trains value/policy
```

That is very noisy and slow.

To get AlphaZero-like improvement, you need some stronger operator than the raw policy:

```text
PUCT
top-k action-value reranking
deeper offline teacher search
tablebase correction
model ensemble
best-response search
```

So the architecture can be search-light at deployment, but the training loop should still include search or reanalysis.

Best pattern:

```text
deployment:
  often no search or tiny search

training:
  use more search than deployment
  then distill it back into the lightweight model
```

This is exactly how to make a lightweight model stronger than its raw supervised version.

## 14. What “scaling” would look like

I would scale along four axes, in this order:

### Axis 1 — better data, not bigger model

```text
more diverse positions
hard-position mining
teacher reanalysis
tablebase correction
cleaner replay buffer
```

### Axis 2 — better search targets

```text
increase PUCT nodes gradually
improve root exploration
store Q/visit/PV distributions
use deeper search only for hard positions
```

### Axis 3 — model capacity

```text
d_model 128 → 192 → 256
layers 4 → 6 → 8 → 12
add richer relation attention
add action-value / uncertainty / multi-ply heads
```

### Axis 4 — deployment compression

```text
FP16 WebGPU
INT8/QAT CPU path
distill larger self-play model into smaller browser model
```

Important: the model used to generate self-play does **not** have to be exactly the deployed model. You can have:

```text
teacher actor:
  larger model + deeper PUCT

deployed student:
  smaller model + search-light inference
```

That is the cleanest way to maximize lightweight strength.

## 15. A practical self-play roadmap

### v0: supervised warm start

```text
Train SquareFormer policy/WDL/action-value on teacher data.
No self-play yet.
```

### v1: shallow self-play

```text
Actors:
  current model
  32-node PUCT
  opening diversity

Train:
  policy = search visits
  value = game result/root WDL
```

Goal:

```text
prove that checkpoint N+1 beats checkpoint N
```

### v2: action-value self-play

```text
Store per-move Q values from PUCT.
Train action-value head on top-k moves.
Use action-value head for move ordering and reranking.
```

Goal:

```text
same strength with fewer nodes, or more strength at same nodes
```

### v3: uncertainty-gated search

```text
Train uncertainty head from:
  value error
  policy/search disagreement
  blunder positions
  teacher disagreement

Use:
  low uncertainty → no/small search
  high uncertainty → more PUCT
```

Goal:

```text
adaptive strength per millisecond
```

```text
Mine:
  losses

  failed puzzles

  deeper PUCT
  Stockfish/lc0
  tablebase
```

Goal:

```text
improve on the model’s actual blind spots
```

### v5: actor/student split

```text
Large actor:
  stronger model/deeper search generates high-quality data

Small student:
  distilled browser model learns actor’s policy/value/action-values
```

```text
strongest lightweight deployed engine
```

## 16. Main risks

### Self-play collapse

The model reinforces its own blind spots.

Mitigation:

```text
teacher anchors
hard-position reanalysis
diverse openings
historical replay
promotion gates
```

### Overfitting to its own search

The net learns artifacts of one search configuration.

Mitigation:

```text
vary node budgets
mix teacher labels
evaluate at multiple search budgets
```

### Value drift

The WDL/value head becomes poorly calibrated.

Mitigation:

```text
game-result targets
root-search WDL targets
tablebase endgames
calibration metrics
```

### Compute waste

Too much search on easy positions.

Mitigation:

```text
uncertainty-gated search
targeted reanalysis
top-k action-value reranking
```

### Search-light overconfidence

The model thinks it is sure when it is wrong.

Mitigation:

```text
uncertainty loss based on actual search error
fallback testing
puzzle suites
teacher disagreement labels
```

## Bottom line

Yes: these architectures can scale into lc0-style self-play **if they preserve the AlphaZero interface**:

```text
policy + value → search → improved policy/value targets → next network
```

The strongest plan is not pure supervised learning and not pure search-less play. It is:

```text
1. Supervised/distilled warm start.
2. Search-improved self-play.
3. Action-value distillation from self-play search stats.
4. Multi-ply auxiliary heads to internalize short continuations.
5. Uncertainty-gated PUCT to spend search only where it matters.
6. Targeted reanalysis of hard positions.
7. Distill larger actor/search into lightweight deployed student.
```

That gives us a credible path to a model that can **improve beyond its initial oracle data** while still optimizing for the real goal: **the strongest lightweight neural chess engine per millisecond, not a dogmatic search-less model.**
