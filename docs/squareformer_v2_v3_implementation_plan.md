# SquareFormer V2/V3 Implementation Plan: Data-First Search-Light Training

## Purpose

This is the concrete implementation plan for turning the current SquareFormer/ChessFormer experiments into a serious `SquareFormer-AV-PUCT` line.

The near-term objective is not architectural novelty for its own sake. It is:

```text
make the training data and labels strong enough that SquareFormer V2 can learn move consequences,
then use V3 to learn tactical risk, uncertainty, and when to spend search.
```

Current lesson from the 100M sweep:

```text
cnn_80x5 is the current searched-strength champion.
ChessFormer/SquareFormer v1 is viable but underpowered/under-informed.
```

Therefore V2 should not be "same transformer, just bigger". V2 should add the missing data and heads:

```text
policy imitation -> candidate action-value/regret ranking -> uncertainty/tactical verification
```

---

## Executive decision: should we label all 100M rows with Stockfish?

Yes, but only for the right label tier.

Do **not** try to generate full action-value labels for every legal move on every one of the 100M rows. That is wasteful and would make the data huge.

Do this instead:

```text
Tier 0: 100M clean supervised policy/WDL rows remain the base.
Tier 1: broad root teacher labels on deduped/bucketed 100M positions, shallow Stockfish MultiPV.
Tier 2: richer candidate action-value labels on a high-yield 5M-10M subset.
Tier 3: deep/tactical labels on 250k-2M hard positions.
Tier 4: on-policy hard-negative labels generated by current best models.
```

Full 100M shallow root labeling is feasible and useful once dedup/sharding is in place. Full 100M candidate-move multi-ply labeling is not the fastest path.

Previous local Stockfish measurement:

```text
3.63M rows, depth 8, MultiPV 4, 16 workers: ~879 rows/sec
100M at same throughput: ~31.6 hours local
```

With cloud CPU workers this becomes a burst job. But the first gain should come from **deduped, bucketed, high-value labels**, not blind brute force.

## North-star model

```text
SquareFormer V2 = SquareFormer-AV
  64 square tokens
  chess relation bias
  policy head
  WDL/value bucket heads
  candidate action-value/regret head
  top-k AV rerank inference

SquareFormer V3 = Tactical/Uncertainty SquareFormer
  V2 trunk/heads
  after-reply value
  opponent-reply policy
  queen/material risk heads
  uncertainty/value-error head
  conditional PUCT fallback
```

V2 answers:

```text
Can the model rank candidate moves by consequence, not just imitate plausible moves?
```

V3 answers:

```text
Can the model identify tactical danger and spend search only when needed?
```

## Position registry and dedupe

Before expensive labeling, build a registry over the 100M supervised rows.

### Registry key

Use a history-aware key for model training, plus a current-position key for teacher dedupe:

```text
position_key = hash(fen board + stm + castling + ep + halfmove bucket)
history_key  = hash(position_key + last 2 history FEN position keys)
row_key      = dataset shard + row offset
```

Stockfish root labels can usually dedupe on `position_key`. Model training joins labels back to rows using `position_key`, while history remains in the model input.

### Registry row schema

```json
{
  "position_key": "sha256:...",
  "history_key": "sha256:...",
  "fen": "...",
  "history_fens": ["..."],
  "played": "e2e4",
  "wdl": [1, 0, 0],
  "source": "lichess_elite_2024-01",
  "shard": "train/shard_0000.jsonl.zst",
  "row": 12345,
  "ply": 18,
  "phase": "opening|middlegame|endgame",
  "material_bucket": "...",
  "opening_key": "first_retained_fen_or_eco_proxy",
  "legal_count": 32
}
```

### Bucket fields to compute now

These make the data much better quickly:

```text
source/month
ply bucket: 10-19, 20-39, 40-79, 80+
phase: opening/middlegame/endgame
material bucket
legal move count bucket
in-check flag
has capture/check/promotion
queen-risk static flag
opening key / ECO proxy
position duplicate count
```

### Why this matters

The current `--skip-plies 10` dataset still contains opening identity and opening bias. A registry lets us cap and sample by opening/source/phase without rebuilding the base 100M every time.

## Teacher label tiers

## Tier 1: broad root Stockfish labels

Purpose:

```text
improve value calibration, policy distribution, and broad tactical quality
```

Recommended first run:

```text
input: deduped/bucketed 25M-100M position_keys
teacher: Stockfish
search: depth 8 or nodes equivalent
multipv: 4 initially, 8 if throughput is acceptable
output: sparse legal policy over MultiPV moves + q/wdl + best move + pv
```

Root label schema:

```json
{
  "position_key": "sha256:...",
  "fen": "...",
  "teacher": "stockfish",
  "teacher_version": "...",
  "depth": 8,
  "nodes": null,
  "multipv": 4,
  "side_to_move_perspective": true,
  "best": "e2e4",
  "q": 0.12,
  "wdl": [0.42, 0.31, 0.27],
  "policy": {"e2e4": 0.45, "d2d4": 0.31, "g1f3": 0.14, "c2c4": 0.10},
  "scores_cp": {"e2e4": 34, "d2d4": 18},
  "pv": {"e2e4": ["e2e4", "c7c5", "g1f3"]},
  "label_time_ms": 42
}
```

Use CP-to-policy conversion:

```text
p(move) ∝ exp((cp(move) - cp_best) / temperature_cp)
temperature_cp: start with 75 or 100 cp
mate scores: clamp/special-case
```

Training use:

```text
policy target = mix(human played one-hot, Stockfish sparse policy)
wdl target    = mix(game result WDL, Stockfish WDL/q bucket)
```

Start conservative:

```text
human_policy_weight = 0.70
teacher_policy_weight = 0.30
result_wdl_weight = 0.70
teacher_wdl_weight = 0.30
```

Do not let shallow Stockfish erase the human/elite policy distribution.

## Tier 2: candidate action-value labels for V2

Purpose:

```text
teach the model which plausible moves survive verification
```

This is the most important V2 dataset.

### First target size

```text
pilot:      100k positions
first real: 1M positions
scale:      5M-10M positions if V2 metrics improve
```

### Candidate selection per position

Do not label all legal moves. Label a compact set:

```text
1. human/played move
2. Stockfish MultiPV top-k, k=8
3. current best CNN policy top-k, k=8 using cnn_80x5
4. current SquareFormer/ChessFormer policy top-k, k=8
5. PUCT top-k from current best model, optional k=4-8
6. all legal checks
7. all captures above SEE/material threshold
8. all promotions
9. queen moves and queen-risk moves
10. random legal distractors, 2-4 moves
```

Deduplicate candidates. Cap at:

```text
max_candidates_per_position = 16 initially
max_candidates_for_tactical = 24
```

### Candidate label schema

```json
{
  "position_key": "sha256:...",
  "fen": "...",
  "teacher": "stockfish",
  "depth": 10,
  "multipv": 8,
  "candidates": [
    {
      "move": "e2e4",
      "reasons": ["played", "teacher_top", "student_top"],
      "cp_after_move": 28,
      "wdl_after_move": [0.42, 0.31, 0.27],
      "best_reply": "c7c5",
      "cp_after_reply": 18,
      "wdl_after_reply": [0.39, 0.32, 0.29],
      "pv": ["e2e4", "c7c5", "g1f3"],
      "regret_cp": 0,
      "q": 0.07,
      "q_after_reply": 0.04,
      "queen_lost_after_reply": false,
      "material_delta_after_reply_cp": 0,
      "tactical_flags": ["quiet"]
    }
  ],
  "best_move": "e2e4",
  "best_cp": 28
}
```

### How to label candidates cheaply

For each candidate move:

```text
position fen F
searchmoves candidate at root -> cp_after_move / PV
make candidate
Stockfish from child -> opponent best_reply and child eval
make best_reply
Stockfish shallow eval or use PV score -> after_reply value
```

For high-throughput first pass, store:

```text
root searchmoves candidate score
teacher best reply from child
after-reply score from child search
```

Do not require perfect multi-ply proof for every candidate in V2. V3 can deepen tactical subsets.

## Tier 3: tactical/deep labels for V3

Purpose:

```text
fix queen/material blunders, tactics, and uncertainty calibration
```

Target sources:

```text
queen-risk suite
model-mined blunders
PUCT512 game failures
puzzle/tactical FENs
positions where cnn_80x5, cnn_64x6, and ChessFormer disagree
positions where shallow/deep Stockfish disagree
positions with large policy/search shift
endgames near tablebase size
```

First target size:

```text
250k deep tactical positions
then 1M-2M if useful
```

Search:

```text
Stockfish depth 12-16 or nodes 50k-200k
MultiPV 8-16
extra best-reply analysis for risky candidate moves
Syzygy exact labels when available
```

Add V3 labels:

```text
queen_lost_after_reply
material_delta_after_reply
best_reply_policy
reply move
value_after_reply bucket
sacrifice_soundness
refutation_probability
uncertainty target = shallow/deep disagreement or model/teacher regret
```

## Tier 4: on-policy hard data

Purpose:

```text
train on the positions our models actually reach and fail in
```

Use current best models as data generators:

```text
cnn_80x5 at policy-only, v32, v128, v512
cnn_64x6 as a contrasting strong CNN
ChessFormer/SquareFormer v1 as transformer failure source
```

Mine from:

```text
model-vs-model arenas
Stockfish/Maia anchor games
browser games
PUCT override diagnostics
queen/material blunder diagnostics
```

Selection rules:

```text
high model confidence but high teacher regret
model top move not in teacher top-k
search overturns policy
large value drop after selected move
queen/material loss after reply
opening lines overrepresented by model preference
```

These rows should be oversampled in V2/V3 training, but kept separate from fixed dev/gates.

## Cloud/distributed labeling plan

Use local workers for pilots and cloud CPU workers for broad Stockfish. Training stays local/centralized.

### Job types

```text
build_registry
root_stockfish_label
candidate_extract
candidate_stockfish_label
tactical_deep_label
validate_shard
merge_manifest
```

### Job manifest

```json
{
  "job_id": "root_sf_d8_mpv4_000123",
  "job_type": "root_stockfish_label",
  "input_uri": "s3/r2/local://positions/shard_0123.jsonl.zst",
  "output_uri": "s3/r2/local://labels/root_sf_d8_mpv4/shard_0123.jsonl.zst",
  "teacher": "stockfish",
  "depth": 8,
  "multipv": 4,
  "threads": 1,
  "hash": "...",
  "row_count_expected": 100000
}
```

### Worker rules

```text
one Stockfish process per worker core or configured slot
write local temp shard first
validate JSON/schema locally
upload/rename atomically to final shard
write sidecar metrics: rows/sec, failures, engine version, command, host
never train on unvalidated shards
```

### Recommended execution

```text
D0 local pilot:
  100k root labels
  100k AV candidate labels
  train tiny V2 smoke

D1 local scale:
  1M root labels
  1M AV labels
  train V2 1M overlay

D2 cloud broad labels:
  25M-100M root labels with CPU workers
  5M-10M AV labels if D1 improves

D3 recurring hard labels:
  250k-2M tactical/deep labels from failures
```

### Cloud sizing intuition

If local 16-worker throughput is ~879 rows/sec for depth8/MultiPV4 style jobs:

```text
100M local:       ~32 hours
4x local CPU:      ~8 hours
16x local CPU:     ~2 hours
```

Candidate AV labels are several searches per position, so scale them by value, not volume.

## SquareFormer V2 implementation

### Architecture changes

Fork or extend `training/train_squareformer_torch.py` into a V2 trainer rather than destabilizing V1 immediately.

New model outputs:

```text
policy logits:         [B, policy_size]
wdl logits:            [B, 3]
q_bucket logits:       [B, q_buckets]
action_value logits:   [B, C, av_buckets] or scalar [B, C]
regret logits/scalar:  [B, C]
```

Candidate move input:

```text
candidates: [B, C, move_features]
move_features:
  from square
  to square
  promotion
  moved piece
  captured piece if known
  check/capture/promotion flags
  queen-risk static flags optional
```

AV head:

```text
h_global = pooled trunk embedding or CLS token
h_from   = hidden[from]
h_to     = hidden[to]
e_move   = embedding(from, to, promotion, move type)
score    = MLP([h_global, h_from, h_to, e_move, tactical_features])
```

Keep relation-bias trunk and from-to policy head initially. Do not change all axes at once.

### V2 losses

Initial loss:

```text
L =
  1.00 * policy_ce_or_KL
+ 1.00 * wdl_ce
+ 0.50 * q_bucket_ce
+ 1.00 * action_value_bucket_ce
+ 0.25 * pairwise_ranking_loss
+ 0.10 * regret_mse_or_huber
```

Use main supervised 100M rows for policy/WDL. Use labeled overlay rows for AV/regret. Mixed batches are acceptable:

```text
normal supervised batch -> policy/WDL/q only
AV labeled batch        -> policy/WDL/q + AV/ranking/regret
```

### V2 inference modes

Add these modes to Node/browser eval:

```text
policy-only
policy + child WDL rerank, k=8/16
policy + AV rerank, k=8/16
PUCT 32/128 with V2 policy/WDL
```

Default V2 experiment:

```text
policy top 12 -> AV rerank -> deterministic best
```

### V2 success gates

V2 is successful if it improves at least one fast mode without collapse:

```text
AV rerank beats policy-only
AV rerank beats child-value rerank
AV rerank reduces teacher regret of selected moves
queen blunder rate does not regress
PUCT strength does not regress
latency increase is acceptable
```

Static metrics:

```text
policy CE/top1/top4/top8
WDL CE/Brier/calibration
q bucket CE
candidate top1 accuracy vs teacher
pairwise ranking accuracy
Kendall tau over candidates
top-k regret cp
```

Arena gates:

```text
policy-only varied openings
AV-rerank varied openings
PUCT 32/128/512 model-vs-model
Stockfish/Maia anchors only after illegal/UCI caveats are controlled
```

## SquareFormer V3 implementation

V3 should start from the best V2 checkpoint and add tactical/uncertainty labels.

### Additional heads

```text
uncertainty/value_error
opponent_reply_policy for candidate moves
after_reply_value_bucket
queen_lost_after_reply
material_delta_after_reply_bucket
tactical_refutation_probability
sacrifice_soundness
```

### V3 losses

```text
L = V2_loss
+ 0.25 * reply_policy_ce
+ 0.25 * after_reply_value_ce
+ 0.25 * queen_lost_bce
+ 0.20 * material_delta_ce_or_huber
+ 0.10 * uncertainty_calibration
+ 0.10 * sacrifice_soundness_ce
```

### Runtime use

```text
if AV confidence high and tactical risk low:
  play AV-reranked move
elif risk/uncertainty medium:
  top-k child verification
else:
  small PUCT 32-128
```

V3 should be measured by:

```text
Elo per millisecond
fallback frequency
fallback precision/recall
queen blunder reduction
sound sacrifice preservation
teacher regret reduction
```

## Promotion gates for V2/V3

A model is not promoted just because supervised CE improves.

Required protocol card:

```text
model artifact hash
dataset manifest hashes
teacher label manifest hashes
training command
backend/runtime
opening suite
search budget
thread settings
illegal counts
latency
```

Minimum gates:

```text
1. policy/WDL non-catastrophic vs V1/CNN baselines
2. AV ranking improves fixed teacher-regret set
3. AV-rerank beats policy-only for same model
4. PUCT 32/128 does not regress badly
5. queen blunder suite improves or holds
6. sound sacrifice preservation holds
7. browser/Node ONNX parity passes
```

Strong promotion gate:

```text
varied-opening paired arena vs cnn_80x5, cnn_64x6, ChessFormer v1
policy-only, AV-rerank, and PUCT 128/512 modes
100+ games for serious claims
```

## What not to do yet

Avoid:

```text
full legal-move action-value labels for all 100M rows
architecture churn before label/schema validation
training only on Stockfish best moves and erasing human policy
hard-banning queen sacrifices
trusting Maia/lc0 anchors until UCI illegal anomalies are hardened
promoting from startpos-only arenas
mixing generated labels into git
```

Prefer:

```text
immutable label overlays
small pilots with inspection
bucketed/deduped scaling
candidate-level action-value/regret
on-policy hard-negative mining
explicit tactical and queen-safety gates
```

---

## Final recommendation

The fastest path to a real SquareFormer V2/V3 is data-first:

```text
1. Build the 100M position registry and bucket reports.
2. Generate broad deduped Stockfish root labels.
3. Generate high-value candidate action-value labels, not all-legal labels.
4. Train V2 with AV/regret ranking and prove AV-rerank beats policy-only.
5. Mine failures and train V3 tactical/uncertainty heads.
6. Use cloud Stockfish workers once the 100k/1M pilots prove label quality.
```

If cloud Stockfish is needed, use it aggressively for validated label jobs. But the multiplier comes from **which positions and candidates we label**, not merely from labeling the largest possible number of rows.
